# JARVIS Codebase Audit — 2026-07-17

Full-codebase "rattlesnake" sweep: seven parallel agents hunted merged-but-broken
wiring, dead code, and drift across seven dimensions (tool wiring, dead code, schema
drift, config drift, jobs, API/UI drift, admitted gaps). Every finding was then
adversarially re-verified by a separate skeptic agent instructed to *refute* it; only
findings confirmed by direct file reads survived. **35 findings confirmed, 3 refuted**
(28 distinct after merging duplicates). Method: multi-agent workflow, 45 agents.

Severity: **critical** = safety gate bypassable or data loss; **high** = a user-facing
feature silently broken, or a security/operational hazard; **medium** = will bite a
future change or a real edge case; **low** = hygiene / doc drift.

Nothing here is on fire in steady state — the app works. These are the dormant snakes.

---

## HIGH — fix soon

### H1. Path-traversal source disclosure in the SPA fallback  🔒 security
`backend/app/main.py:316` — The catch-all that serves the React app does
`FileResponse(os.path.join(static_dir, full_path))` guarded only by
`full_path.startswith("api/")`. There is no `..` / normpath containment check. A
non-normalizing request like `/..%2fapp%2fconfig.py` resolves to `/app/app/config.py`
and returns real backend source — `config.py`, `totp.py`, etc. (Reproduced against the
project's own Starlette version.) The `/assets` mount uses `StaticFiles`, which has this
protection built in; this hand-rolled handler does not.
**Fix:** resolve the candidate path and verify `os.path.commonpath` keeps it inside the
static dir, else return the SPA index (or 404).

### H2. Migration chain cannot bootstrap a fresh production database
`backend/alembic/versions/0010_flight_booking.py:47` (and `0006_task_google_id.py`) —
Nine tables (`users`, `conversations`, `messages`, `contacts_whitelist`,
`pending_confirmations`, …) are created only by `create_all()` in the **api** process
lifespan, never by a migration. Migrations `0006` and `0010` then `add_column` onto
`tasks` / `pending_confirmations` guarded only by dialect, not table existence. Fly's
`release_command = "alembic upgrade head"` runs **before any machine boots**, so on a
fresh/restored Postgres it hits `UndefinedTable` and aborts the deploy before
`create_all` could ever run. **Disaster recovery and any second environment are broken
today.** (Related: **M1**.)
**Fix:** add a real baseline migration that creates the nine bootstrap tables (guarded
with `to_regclass`, the pattern `0001` already uses correctly), or make the two
`add_column`s table-existence-guarded and add the base tables to the chain.

### H3. Unanswered outbound calls get stuck "ringing" forever
`backend/app/routes.py:287` — `/api/voice/status` only updates the `OutboundCall` row
when `CallStatus == "completed"`. Twilio fires the status callback on *any* terminal
state (no-answer / busy / failed / canceled), but those fall through unhandled. Result:
if you don't pick up a call-me-back, the row stays `ringing` forever — `due_calls` only
re-dials `queued`, so the callback is silently lost; `pending_callbacks` lists the ghost
row forever; and `cancel_callback` refuses it (it only cancels `queued`). The
`"no_answer"` close-out at line 296 is unreachable.
**Fix:** branch on the actual `CallStatus`; mark unanswered calls `no_answer` (and decide
retry vs. give-up), so the row leaves `ringing`.

### H4. `voice_enabled` kill switch is inert
`backend/app/config.py:66` — Defined (default `False`), documented in `ARCHITECTURE.md`
as a kill switch, set in `.env.template` — but **read by zero code**. The voice webhooks
are mounted unconditionally and check only Twilio signature + caller-ID whitelist. An
operator who sets `VOICE_ENABLED=false` to shut off the spoofable voice channel changes
nothing. Every *other* documented kill switch (`enable_trading`, `booking_enabled`,
`outbound_calls_enabled`, …) is actually enforced; this one alone is a false control.
**Fix:** honor the flag in the voice routes (return empty TwiML / 403 when off), or drop
it and update the docs.

### H5. `forget_episode` can delete the wrong memory
`backend/app/handlers/episodes.py:38` — `forget_episode` requires an `episode_id`, and
its own description says "get the id from `recall_episodes`" — but `_format_episode`
(used by both `recall_episodes` and `recall`) prints date/title/summary/quotes and
**never the id**. So the model must guess an integer. Since ids are sequential, a guess
can silently delete the wrong episode (cascade-deleting its quotes), and there's no
confirmation gate. The fact store gets this right (`recall_facts` prints `#id:`).
**Fix:** include the id in `_format_episode` output (or add an `about:` text selector
with disambiguation, like `forget_fact`).

### H6. Tailscale capability is silently unroutable
`backend/app/agents.py:145` — `tailscale_status` is registered and works, and it's in
the `netstatus` agent's roster — but that agent's **description**, which is the only
routing signal the orchestrator sees, explicitly says "…NOT Tailscale." So tailnet
questions never reach the tool; JARVIS says she can't see it. And `seed_agents`
overwrites the DB description from code on every deploy, so fixing it in the Admin UI
reverts. This is the exact failure class the code's own comments document for a prior
`sync_google_contacts` bug.
**Fix:** update the `netstatus` description (and system prompt) to advertise Tailscale.

### H7. The "watch for X" feature is silently unroutable
`backend/app/agents.py:107` — `watch_for` / `list_watches` / `cancel_watch` live only in
the `secretary` roster, but neither her description nor system prompt mentions
watching/alerting. So "call me if rpi-02 goes down" routes to `netstatus` (no watch
tools) and degrades to a one-time status check instead of a standing watch. Same
description-is-the-router root cause as H6.
**Fix:** add watch/alert language to the `secretary` description, or split watches into
their own advertised capability.

## MEDIUM — fix opportunistically

- **M1. ✅ RESOLVED (PR #9).** Nine core tables have no create migration (`models.py`) —
  the structural root of **H2**; the `0013_baseline` migration resolved it.
- **M2. ✅ RESOLVED (PR #17).** Orphaned jobs are never recovered (`jobs.py`) — a job
  claimed as `running` when the worker dies or Fly redeploys was never reset to `queued`
  and was silently lost. Now `recover_stale_jobs()` (a staleness-windowed sweep in the
  worker loop) re-queues stranded jobs, or fails them if past `max_attempts`.
- **M3. ✅ RESOLVED (PR #17).** Briefing error text gets read aloud (`jobs.py`) — the
  exception fallback slipped past the `(no briefing` guard, so a briefing *call* read the
  error + raw data dump aloud. Degraded outputs now carry a sentinel and
  `is_speakable_briefing()` gates the call path.
- **M4. ✅ RESOLVED (PR #17).** Watch consumed even when the call can't be placed
  (`watches.py`) — a one-shot watch was marked `done` even when `schedule_call` returned
  `None`. Now the watch stays active and retries on its next check when the alert can't be
  delivered.
- **M5. ✅ RESOLVED (PR #18).** `owner_passport` is never read (`config.py`) — the Duffel
  passenger builder omitted passport data, so international bookings failed *after* the
  confirmation gate. Now `_passenger_from_whoami(include_passport=…)` attaches a Duffel
  `identity_document` for international offers (which need number + expiry + issuing
  country — two new settings, `owner_passport_expiry` / `owner_passport_country`), the
  pregate **refuses an intl fare up front** if any are missing, and `explain_duffel_error`
  has a passport branch as a backstop. Domestic offers never send a passport.
- **M6. ✅ RESOLVED — descoped (PR #24).** Two of four `user_tz_source` precedence entries
  (`stated`, `active_trip`) were never implemented, over-promising the contract.
  `_resolve_user_tz`'s docstring and TDD §4.1 now document only the live sources
  (`location_report` > `default`) and mark the two richer sources as future work in their
  own TDDs (they're genuine mini-features: tz extraction/mapping/TTL, and
  `Trip.destination_tz`). No behavior change — the phone location report already covers the
  common case.
- **M7. ✅ RESOLVED — descoped (PR #24).** The "outside the request window" date flag was
  never wired (needs the caller to infer a window). Removed the dead `window_start`/
  `window_end` parameters and the unreachable `[outside window: …]` branch from
  `flag_stale_dates`; kept the working past-date `[stale:]` flagging; TDD §4.3 marks
  request-window flagging as future work.

**All 8 MEDIUM findings resolved.**
- **M8. ✅ RESOLVED (PR #9).** Dead voice-confirmation constants (`voice_pipeline.py`) —
  the unreferenced `VOICE_AFFIRMATIVE`/`VOICE_NEGATIVE` duplicates were removed; the live
  gate's `orchestrator._VOCAB` is now the single source of truth.

**Remaining after 2026-07-17:** none at MEDIUM — M6 and M7 were formally descoped (the
`stated`/`active_trip` timezone sources and request-window date flagging are future TDD work;
the dead code and over-promises were removed). All HIGH and MEDIUM findings are resolved.

## LOW — hygiene & doc drift

- **L1. `place_calls` job handler is orphaned** (`jobs.py:284`) — nothing enqueues it;
  the live path is `_place_due_calls` in the worker. `ARCHITECTURE.md` and the 4am-call
  doc still point at the dead handler. Remove it; fix the docs.
- **L2. `close_episode` dedupe is a latent text-channel bug** (`episodic.py:60`) — it
  matches *any* prior `distill_episode` job for the `thread_key`, ignoring `source_ref`.
  Safe now because call-sids are unique; the moment text channels adopt this (as the
  docstring plans), every conversation after the first on a phone-number thread is never
  distilled.
- **L3. Literal `\n` in the watch-judge prompt** (`watches.py:211,218`) — `"\\n"` in a
  non-raw string sends the two characters, not a newline. Haiku tolerates it; cosmetic.
- **L4. `amadeus_api_key` is dead config** (`config.py:217`) — only Duffel is wired.
- **L5. `LoginRequest` schema is dead** (`schemas.py:7`) — login uses
  `OAuth2PasswordRequestForm`.
- **L6. `_plural()` is a dead helper** (`netstatus.py:80`) — `_uptime()` re-inlines the
  same logic instead.
- **L7. `memory_embeddings` dimension hardcoded** (`alembic/0001:76`) — migration pins
  `vector(1024)` while runtime builds from `settings.embedding_dim`; changing the
  embedding model with pgvector on would error at insert. Dormant (pgvector defaults off).
- **L8. `Episode.embedding` docstring drift** (`models.py:387`) — claims a "pgvector
  mirror in prod" that doesn't exist for episodes (only for facts).
- **L9. `/api/conversations` + messages have no UI consumer** (`routes.py:434`) —
  documented future surface; authenticated and harmless.
- **L10. `/api/memory/audit` has no UI entry point** (`routes.py:397`) — its docstring
  promises browser readability, but `OAuth2` bearer auth means it can't be opened by URL
  and nothing fetches it. Still reachable via the `audit_memory` email tool.
- **L11. `/status` page + tab title are scaffold boilerplate** (`ui/src/pages/DashboardPage.jsx:28`,
  `ui/index.html:6`) — the logged-in Status tab still shows "Replace this page with your
  app…", and the browser tab still reads "Vite + FastAPI Starter". The TDD calls the
  title a five-minute must-do.
- **L12. Briefing footer contradicts itself** (`briefing.py:27`) — always says "Not yet
  connected: … Weekend & travel …" even while it now renders a live `## Travel` section.
- **L13. Action-item→task linking not wired** (`models.py:156`) — known, documented in the
  episodic TDD: `Task` has no `thread_key`, so call action-items can't link to tasks.

## Refuted (reported by a hunter, killed by verification)

- `/api/conversations`, `/api/jobs`, `/api/memory/audit` as "dead API" — **not dead**:
  documented, tested, operator/browser-facing surface by design (only the *UI wiring* is
  absent, captured as L9/L10).

  helper; TOTP enrollment already happened and the factor is live.
